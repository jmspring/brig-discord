# Multi-instance manifest and service support

## Goal

Document and support the pattern for running multiple brig-discord instances as
separate brig persistent skills, each with its own manifest, rc.d service, jail,
secret, and gateway identity.

## Context

After the configurable gateway name ticket (bd-xj85), brig-discord supports
`BRIG_GATEWAY_NAME` and `BRIG_SESSION_PREFIX` env vars. But to actually deploy two
Discord bots, the operator needs two separate persistent skill registrations in brig,
each with its own manifest, secret, and service name.

The current `manifest.toml` is a single file with hardcoded values. This ticket creates
a pattern for multi-instance deployment.

**Depends on:** "Configurable gateway name and scope" (bd-xj85).

## Required Changes

### 1. Create `contrib/` directory with example manifests

Create example manifests for multi-instance deployment:

`contrib/manifest-ops.toml`:
```toml
[skill]
name = "discord-ops-bot"
description = "Discord gateway for ops team"
kind = "persistent"

[requires]
network = ["discord.com", "gateway.discord.gg"]
max_runtime = "forever"

[persistent]
rc_name = "brig_disc_ops"
entrypoint = "/usr/local/bin/brig-discord"
restart_on_failure = true
depends_on = ["brig"]
env = { BRIG_GATEWAY_NAME = "discord-ops-bot", BRIG_SESSION_PREFIX = "disc-ops" }

[persistent.socket]
capabilities = ["submit_task", "read_status"]

[secrets]
discord_token = { env = "BRIG_DISCORD_TOKEN" }
```

`contrib/manifest-community.toml`:
```toml
[skill]
name = "discord-community-bot"
description = "Discord gateway for community server"
kind = "persistent"

[requires]
network = ["discord.com", "gateway.discord.gg"]
max_runtime = "forever"

[persistent]
rc_name = "brig_disc_community"
entrypoint = "/usr/local/bin/brig-discord"
restart_on_failure = true
depends_on = ["brig"]
env = { BRIG_GATEWAY_NAME = "discord-community-bot", BRIG_SESSION_PREFIX = "disc-community" }

[persistent.socket]
capabilities = ["submit_task", "read_status"]

[secrets]
discord_token = { env = "BRIG_DISCORD_TOKEN" }
```

### 2. Verify brig's persistent skill `env` support

The manifests above use `[persistent] env = { ... }` to pass environment variables to
the entrypoint. Verify that brig's skill enable/rc.d generation actually reads and
injects these env vars. If not, this is a dependency on a brig-side change (document
it but don't block on it — operators can set env vars in the rc.d script manually).

### 3. Document multi-instance deployment in docs/GUIDE.md

Add a "Running Multiple Bots" section to the guide covering:

1. Register each bot in the Discord Developer Portal (different application, different token)
2. Each bot needs its own: application, bot token, OAuth2 invite URL, server invite
3. Create a manifest per bot (copy and customize from contrib/)
4. Install the single binary once (`/usr/local/bin/brig-discord`)
5. Register each manifest as a separate skill:
   ```sh
   brig skill add --manifest contrib/manifest-ops.toml
   brig skill add --manifest contrib/manifest-community.toml
   ```
6. Set secrets for each:
   ```sh
   brig secret set discord-ops-bot.discord_token
   brig secret set discord-community-bot.discord_token
   ```
7. Enable and start each:
   ```sh
   sudo brig skill enable discord-ops-bot
   sudo brig skill enable discord-community-bot
   sudo service brig_disc_ops start
   sudo service brig_disc_community start
   ```
8. Explain that each bot has isolated memory in brig (facts scoped to its gateway name)
9. Explain that session keys are distinct (different prefix)
10. Note: each bot must be separately invited to its target Discord server(s)
